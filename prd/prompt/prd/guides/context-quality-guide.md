# Context 작성 품질 가이드

## 전체 원칙

**Context 파일의 성격**:
- 초기 버전은 Why 문서 기반으로 작성
- Feature PRD 작성 과정에서 점진적으로 구체화되고 보완됨
- 모든 Feature PRD가 이 파일을 참조하므로 일관성 유지가 중요

**작성 수준**:
- Core User Actions는 추상적 수준으로 작성 (구체적 UI 구현 세부사항 제외)
- Feature PRD 단계에서 구체적 기능으로 전환됨

**숫자 제한 없음**:
- 에픽 목적 달성에 필요한 만큼 충분히 작성
- 가이드라인의 숫자는 참고용이며 강제 사항이 아님

---

## Context 섹션

### 구조
- 에픽(화면/플로우) 진입 상황을 충분히 서술하는 내러티브 구성
- 첫 번째 문단: 사용자 상황 + 질문들
- 두 번째 문단: Why 문서 연결 + Pain point
- 필요 시 추가 문단으로 맥락 보완

### 필수 요소
- **사용자 질문**: 인용문 형태로 포함 (일반적으로 3-5개)
  - 예: "현재 몇 개 기업의 온라인 부스가 만들어져 있지?", "어떤 기업을 공개하고 어떤 기업을 비공개로 둘지 어떻게 정하지?"
- **Why 문서 연결**: 핵심 문제를 에픽 맥락으로 연결
  - 예: "홈페이지의 포트폴리오 섹션이 제대로 관리되지 않아 고객이나 파트너사에게 보여줄 체계적인 자산이 필요하다는 인식을 갖고 있다"
- **Pain Point**: 구체적 비즈니스 상황으로 표현
- **동기와 목표**: 원하는 것 + 피하고 싶은 것
- **제약 조건**: 시간/리소스/역량 한계 암시

## User Situation 섹션

### 구조
- 불릿 포인트 형식으로 사용자 상황과 니즈 작성
- 에픽 목적에 필요한 만큼 충분히 작성 (일반적으로 4-6개)

### 필수 포함 요소
1. **Why 문서 연결**: 문제 해결 결과 + 현재 상태
   - 예: "Host는 100시간 이상 걸리던 수작업을 AI로 자동화했지만, 생성된 온라인 부스들을 체계적으로 관리할 시스템이 필요하다"

2. **욕구의 대비**: 원하는 것 vs 원하지 않는 것
   - 예: "투자한 모든 기업을 공개하고 싶지는 않으며, 대표적인 성공 사례를 먼저 보여주고 싶어 한다"

3. **Pain Point**: 구체적 비즈니스 문제 + 해당 에픽 필요성
   - 예: "홈페이지 포트폴리오 섹션이 방치되어 있어, 고객과 파트너에게 투자 안목을 증명할 체계적인 쇼케이스가 필요하다"

4. **제약 조건**: 시간/리소스/역량 한계
   - 예: "바쁜 업무 중에도 포트폴리오 상태를 빠르게 파악하고 필요한 관리 작업을 즉시 수행해야 한다"

## Core User Actions 섹션

### 작성 수준 (중요)
**추상적 행동 목록으로 작성**:
- Feature PRD 단계에서 구체적 기능으로 전환됨
- UI 구현 세부사항은 포함하지 않음
- 사용자의 의도와 목표 중심으로 작성

### 구조
- 에픽 목적 달성에 필요한 만큼 작성 (일반적으로 5-8개)
- 각 행동은 독립적이고 실행 가능한 단위

### 작성 원칙
- **추상적 표현**: 구체적 UI보다는 사용자 행동의 본질 표현
  - 좋은 예: "투자 이력 확인", "관심 기업 선택", "포트폴리오 상태 파악"
  - 나쁜 예: "드래그앤드롭으로 순서 조정", "토글 버튼 클릭"
- **비즈니스 가치 명확**: 행동의 목적이 드러나도록 작성
  - 예: "외부 공개 기업 순서 조정" (어떻게가 아닌 무엇을)
- **명확한 동사**: "~을 확인한다", "~을 선택한다", "~을 파악한다"
- **에픽 Goal 연결**: 에픽 목적과 직접 연결되는 핵심 행동

### Feature로 전환
- Feature PRD 작성 시 이 추상적 행동들이 구체적 기능으로 전환됨
- 하나의 Action이 여러 Feature가 될 수도, 여러 Action이 하나의 Feature가 될 수도 있음
- 필요 시 Feature PRD 작성 중 Core User Actions를 추가하거나 수정 가능

## Transitions 섹션

### 작성 원칙
- 버튼명 또는 구체적 액션 명시
- 이동할 에픽(Epic N 또는 `{N}-{epic-name}`)과 제목 명시
- 예: "'포트폴리오 생성' 버튼 클릭 → Epic 2 (포트폴리오 생성)"

## 품질 검증 체크리스트

### Context 품질
- [ ] 에픽(화면/플로우) 진입 상황을 충분히 서술하는 내러티브 제공
- [ ] 사용자 질문이 인용문 형태로 포함됨
- [ ] Why 문서의 핵심 문제가 자연스럽게 연결
- [ ] Pain point가 구체적 비즈니스 상황으로 표현
- [ ] 원하는 것과 피하고 싶은 것 모두 포함

### User Situation 품질
- [ ] 사용자 상황과 니즈가 불릿 포인트로 충분히 작성됨
- [ ] Why 문서의 문제 해결 결과와 현재 상태 연결
- [ ] 원하는 것 vs 원하지 않는 것의 대비 명확
- [ ] 구체적 비즈니스 pain point와 해당 에픽 필요성 드러남
- [ ] 사용자 제약 조건 언급

### Core User Actions 품질
- [ ] 에픽 목적 달성에 필요한 만큼 충분히 작성됨
- [ ] **추상적 수준으로 작성** (UI 구현 세부사항 제외)
- [ ] 각 행동이 사용자의 의도와 목표를 명확히 표현
- [ ] 비즈니스 가치가 명확
- [ ] 명확한 동사 사용
- [ ] 에픽 Goal과 직접 연결

### Transitions 품질
- [ ] 버튼명 또는 구체적 액션 명시
- [ ] 이동할 에픽(Epic N 또는 `{N}-{epic-name}`)과 제목 정확히 기재

### 전체 일관성
- [ ] Context는 점진적으로 구체화될 수 있음을 인지
- [ ] Feature PRD 작성 중 보완 필요 시 수정 가능함을 인지
- [ ] 모든 Feature PRD가 이 Context를 참조함을 인지
