# Feature 1-1: 사진 목록 확인 [필수]

> 📄 **Context**: [1-photo-management-context.md](./1-photo-management-context.md)

## Scenario Context

**Epic**: Epic 1 - 사진 관리 (`1-photo-management`)

**Purpose**: 사진 목록을 보고, 업로드·삭제·순서 변경을 통해 포트폴리오 사진을 직접 관리한다.

**User Situation**: 개발자를 거치지 않고 사진을 올리고, 삭제하고, 순서를 바꾸고 싶어 한다. 여러 장을 한 번에 처리하고 직관적인 조작을 원하며, 포트폴리오에 반영된 이미지는 최적화(1MB 이하, webp 등)가 적용되어 있어야 한다. 모바일에서도 사진 확인이 가능해야 한다.

**Core User Actions** (본 feature와 관련):

- 현재 포트폴리오에 반영된 사진 목록을 확인한다.
- 필요 시 사진별 캡션(파일명 기반) 정보를 인지한다.

---

## User Story

As a **포토그래퍼(친구)** I want to **현재 포트폴리오에 반영된 사진 목록을 순서대로 확인한다**, so that **어떤 사진이 올라가 있는지, 어떤 순서로 노출되는지 파악하고 이후 업로드·삭제·순서 변경을 할 수 있다**.

### Acceptance Criteria

- [x] 로그인된 사용자가 사진 관리 화면에 진입하면 현재 포트폴리오에 반영된 사진 목록이 노출 순서대로 표시된다.
- [x] 각 사진은 썸네일(또는 미리보기)과 캡션(파일명 기반, 예: "이름, 연도"에서 `(N)` 제거)을 함께 보여준다.
- [x] 목록은 **갤러리 구조**(그리드 레이아웃, 썸네일 + 캡션)로 표시하며, 모바일·데스크톱 모두에서 확인 가능하고 shadcn 지침에 맞는 스타일을 사용한다.
- [x] 목록이 비어 있을 때(사진 0장) Empty State가 표시된다.
- [x] 목록 로딩 중에는 Loading State가, 오류 시에는 Error State가 표시된다.

### User Flow

1. 사용자가 Admin 로그인 후 사진 관리 화면에 진입 → 시스템이 현재 포트폴리오 사진 목록(순서 포함)을 조회한다.
2. 조회 성공 → **갤러리 형태**로 노출 순서대로 표시(그리드 셀: 썸네일 + 캡션). 이후 업로드·삭제·순서 변경 기능으로 이어질 수 있다.
3. 조회 실패 → Error State 표시, 재시도 유도.

**구현 참고**: 현재 구조에서는 `images_dir.ts`와 같은 순서 정보와 R2(또는 동일 스토리지)의 파일 목록이 소스이다. Admin에서는 이 순서를 읽어와 **갤러리(그리드, 썸네일+캡션)** 로 표시하고, 이후 Feature에서 순서 변경 시 해당 소스를 갱신하는 방식으로 구현할 수 있다.

### UI State

- **목록 레이아웃**: 갤러리 그리드(여러 열에 걸쳐 썸네일 + 캡션). 반응형 그리드(모바일 1~2열, 데스크톱 3~4열), 호버 시 썸네일 강조.
- **Empty State**: 사진이 한 장도 없을 때. "아직 등록된 사진이 없습니다. 사진을 업로드해 보세요." 등 문구와 업로드 유도(업로드 Feature와 연결).
- **Loading State**: 목록 조회 중 스켈레톤 갤러리 그리드 또는 스피너 표시.
- **Error State**: 목록 조회 실패(네트워크, 권한 등) 시 안내 메시지와 재시도 버튼 표시.
