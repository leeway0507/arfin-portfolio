## Overview

- 기존 feature.tsx 파일을 spec.md에 명시된 디자인 스펙에 맞춰 리팩토링한다.
- 시각적 스타일링(CSS/UI)에만 집중하며, 기능 로직 변경은 배제한다.
- 디자인 토큰(색상, 간격)을 스펙에 명시된 값과 정확히 일치시킨다.

## Input

- **기존 feature.tsx 파일**: 리팩토링 대상이 되는 현재 구현 파일
- - spec.md 파일: 준수해야 할 디자인 스펙 문서
  * 디자인 토큰(색상, 간격, 타이포그래피 등) 정의
  * UI 컴포넌트 스타일 가이드라인

## Output

- **리팩토링된 feature.tsx 파일**: spec.md의 디자인 스펙을 준수하는 코드
- **품질 기준**:
  - 스펙에 명시된 디자인 토큰을 정확히 사용
  - 시각적 스타일링만 변경, 기능 로직은 유지
  - 기존 기능 동작 보장

## Step

**중요: 각 단계는 반드시 순차적으로 완료해야 한다. 한 번에 여러 단계를 진행하지 않는다. 각 단계 완료 후 중간 결과를 확인하고 다음 단계로 진행한다.**

### 1단계: 스펙 분석

**실행 규칙:**
- 이 단계만 먼저 완료한다. 다른 단계로 진행하지 않는다.
- 완료 후 사용자에게 보고하고 다음 단계로 진행한다.

**작업 내용:**
- spec.md 파일을 읽고 디자인 요구사항 파악
- 디자인 토큰 목록 추출(색상 코드, 간격 값, 타이포그래피 등)
- 추출한 디자인 토큰 목록을 정리하여 사용자에게 보고

**완료 조건:**
- 스펙 파일 분석 완료
- 디자인 토큰 목록 추출 완료
- 사용자에게 1단계 완료 보고

### 2단계: 현재 코드 분석

**실행 규칙:**
- 1단계가 완료된 후에만 시작한다.
- 완료 후 사용자에게 보고하고 다음 단계로 진행한다.

**작업 내용:**
- 기존 feature.tsx 파일의 현재 스타일링 구조 파악
- Tailwind CSS 클래스 사용 현황 확인
- 변경이 필요한 스타일 요소 목록화
- 변경 대상 요소 목록을 사용자에게 보고

**완료 조건:**
- 현재 코드 분석 완료
- 변경 필요 요소 목록 작성 완료
- 사용자에게 2단계 완료 보고

### 3단계: 스타일 리팩토링

**실행 규칙:**
- 2단계가 완료된 후에만 시작한다.
- 한 번에 모든 스타일을 변경하지 않고, 변경 대상 요소를 하나씩 처리한다.
- 각 변경 후 Lint 검사를 수행한다.
- 완료 후 사용자에게 보고한다.

**작업 내용:**
- 스펙의 디자인 토큰을 Tailwind CSS 클래스로 매핑
- 변경 필요 요소 목록에서 하나씩 처리한다.
  - 각 요소의 스타일을 스펙에 맞게 교체
  - 기능 로직 코드는 변경하지 않음
  - 변경 후 `read_lints`로 오류 확인 및 수정
- 모든 요소 변경 완료 후 사용자에게 보고

**완료 조건:**
- 모든 스타일 리팩토링 완료
- Lint 오류 없음
- 기능 로직 변경 없음 확인
- 사용자에게 3단계 완료 보고

### 4단계: 검증

**실행 규칙:**
- 3단계가 완료된 후에만 시작한다.
- 최종 검증만 수행한다.

**작업 내용:**
- 스펙의 모든 디자인 토큰이 정확히 적용되었는지 확인
- 기능 로직이 변경되지 않았는지 확인
- 시각적 스타일링만 변경되었는지 최종 점검
- 최종 Lint 검사를 수행한다.
- 사용자에게 작업 완료를 보고한다.

**완료 조건:**
- 모든 디자인 토큰 적용 확인 완료
- 기능 로직 변경 없음 확인
- 최종 Lint 검사 통과
- 사용자에게 작업 완료 보고

## 실행 원칙

### 단계별 진행 규칙

1. **순차 실행**: 각 단계는 반드시 이전 단계가 완료된 후에만 시작한다.
2. **중간 확인**: 각 단계 완료 후 결과를 확인하고 사용자에게 보고한다.
3. **점진적 변경**: 3단계에서는 한 번에 모든 스타일을 변경하지 않고 요소별로 처리한다.
4. **검증 필수**: 각 단계 완료 후 다음 단계로 진행하기 전에 완료 조건을 확인한다.
5. **로직 보존**: 기능 로직은 절대 변경하지 않는다. 시각적 스타일링만 변경한다.

### 금지 사항

- 한 번에 여러 단계를 동시에 진행하지 않는다.
- 단계 완료 전에 다음 단계로 진행하지 않는다.
- 중간 보고 없이 작업을 진행하지 않는다.
- 기능 로직을 변경하지 않는다.
- 한 번에 모든 스타일을 변경하지 않는다.
